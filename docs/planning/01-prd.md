## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

학원에서 새 학생이 오면, 선생님이 종이 시험지를 인쇄해서 단어를 물어보고, 손으로 채점하고, 노트에 결과를 적습니다. 마치 매번 수동으로 체온을 재고 수기로 기록하는 것과 같습니다. 학생이 10명이면 괜찮지만, 30명, 50명이 되면 시간이 엄청나게 걸리고, 이전 결과를 찾아보는 것도 힘듭니다.

### 1.2 목표 상황 (To-Be)

학생이 컴퓨터나 핸드폰에서 단어 테스트를 풀면, 자동으로 채점되고 레벨이 판정됩니다. 선생님은 한 화면에서 모든 학생의 레벨을 한눈에 확인할 수 있습니다. 마치 자동 체온계처럼, 측정하면 바로 결과가 나오고 기록이 남는 것과 같습니다.

### 1.3 왜 지금인가? (Why Now)

학원 운영에서 학생 레벨 파악은 매일 일어나는 일입니다. 종이 테스트의 번거로움이 누적되면서 선생님의 업무 부담이 커지고, 체계적인 학생 관리가 어려워지고 있습니다. 디지털 도구로 이 반복 작업을 자동화하면, 선생님은 수업 준비와 학생 지도에 더 집중할 수 있습니다.

## 2. 사용자 페르소나

### 2.1 주요 페르소나 A: 선생님

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "꼼꼼한 민수 선생님" |
| 나이/직업 | 30대 영어 학원 강사 |
| 핵심 니즈 | 학생들의 단어 수준을 빠르게 파악하고 체계적으로 관리 |
| 불편 포인트 | 종이 테스트 채점, 결과 기록, 레벨 판정에 시간이 너무 많이 걸림 |
| 사용 상황 | 신규 학생 등록 시, 정기 레벨 테스트 시 (학원 PC 사용) |

### 2.2 주요 페르소나 B: 학생

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "열심히 하는 지은이" |
| 나이/직업 | 중학생, 영어 학원 수강생 |
| 핵심 니즈 | 자기 영어 단어 실력을 알고, 부족한 부분을 채우고 싶음 |
| 불편 포인트 | 종이 시험은 지루하고, 결과를 바로 알 수 없음 |
| 사용 상황 | 학원 수업 전/후, 집에서 핸드폰으로 테스트 (모바일 사용) |

### 2.3 페르소나의 하루 (시나리오)

**민수 선생님의 하루:**
오전에 신규 학생 3명이 등록했습니다. 선생님은 PC에서 "단어 레벨 테스트"를 열고, 학생들에게 접속 링크를 알려줍니다. 학생들이 핸드폰으로 테스트를 푸는 동안 선생님은 다른 수업 준비를 합니다. 10분 후, 3명의 테스트 결과가 자동으로 나옵니다. "지은이는 레벨 3, 준호는 레벨 2, 서연이는 레벨 4" — 바로 적절한 반에 배치할 수 있습니다.

**지은이의 하루:**
수업이 끝나고, 선생님이 "이번 주 테스트 해보세요"라고 합니다. 집에 돌아와 핸드폰으로 접속해서 테스트를 시작합니다. 단어가 나오면 뜻을 선택하고, 틀리면 "아쉬워요! 다음엔 맞출 수 있어요!"라는 메시지가 나옵니다. TTS 버튼을 누르면 원어민 발음도 들을 수 있어서 좋습니다. 테스트가 끝나면 "레벨 3 → 레벨 4 달성! 축하해요!"라는 결과가 나옵니다.

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩/로그인)
- [ ] 선생님으로서, 간편하게 계정을 만들 수 있어야 한다. 왜냐하면 복잡한 가입 절차는 도구 도입을 막기 때문이다.
- [ ] 선생님으로서, 학생 계정을 간편하게 생성/관리할 수 있어야 한다. 왜냐하면 학생이 직접 가입하기 어려울 수 있기 때문이다.
- [ ] 학생으로서, 선생님이 알려준 정보로 쉽게 로그인할 수 있어야 한다. 왜냐하면 로그인이 어려우면 테스트 자체를 포기하기 때문이다.
- [ ] 사용자로서, 로그인 상태가 유지되어야 한다. 왜냐하면 매번 로그인하는 건 귀찮기 때문이다.

### FEAT-1: 단어 레벨 테스트 (MVP)
- [ ] 학생으로서, 화면에 나오는 단어의 뜻을 선택할 수 있어야 한다. 왜냐하면 이것이 레벨을 판정하는 핵심 활동이다.
- [ ] 학생으로서, 테스트 중 단어의 발음을 들을 수 있어야 한다(TTS). 왜냐하면 발음을 아는 것도 단어 학습의 일부이다.
- [ ] 학생으로서, 테스트 완료 후 바로 결과(레벨)를 확인할 수 있어야 한다. 왜냐하면 즉각적인 피드백이 동기를 유지시키기 때문이다.
- [ ] 선생님으로서, 학생별 테스트 결과를 한눈에 볼 수 있어야 한다. 왜냐하면 이것이 반 배치와 수업 계획의 기초 자료이기 때문이다.
- [ ] 선생님으로서, wordtest.xlsx 데이터를 기반으로 테스트 문제가 자동 생성되어야 한다. 왜냐하면 직접 문제를 만드는 건 시간이 많이 걸리기 때문이다.

### FEAT-2: 결과 리포트/통계 (v2 이후)
- [ ] 선생님으로서, 학생별/반별 레벨 변화 추이를 그래프로 볼 수 있어야 한다.
- [ ] 선생님으로서, 가장 많이 틀리는 단어 목록을 확인할 수 있어야 한다.
- [ ] 학생으로서, 나의 레벨 변화 히스토리를 확인할 수 있어야 한다.

### FEAT-3: 단어 학습 모드 (v2 이후)
- [ ] 학생으로서, 테스트에서 틀린 단어를 반복 학습할 수 있어야 한다.
- [ ] 학생으로서, 플래시카드 형태로 단어를 공부할 수 있어야 한다.
- [ ] 학생으로서, 학습 후 재테스트로 실력 향상을 확인할 수 있어야 한다.

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- FEAT-1: 단어 레벨 테스트 (wordtest.xlsx 기반)
- FEAT-0: 기본 온보딩/로그인 (선생님 계정 + 학생 계정 관리)
- TTS 발음 듣기 기능

### 5.2 옵션 (시간 허용 시)
- 테스트 결과 히스토리 보기 (간단 버전)
- 선생님 대시보드 (학생 목록 + 최근 테스트 결과)

### 5.3 Non-goals (명시적 제외)
- FEAT-2 결과 리포트/통계 — v2에서 구현 (MVP에서는 기본 결과만 표시)
- FEAT-3 단어 학습 모드 — v2에서 구현 (MVP에서는 테스트만)
- 다크 모드 — MVP에서는 라이트 모드만
- 수익화 기능 — 자체 학원 사용 목적
- 소셜 로그인 — 자체 계정으로 충분
- 알림/푸시 기능 — v2에서 검토

## 7. 해결 가설과 실험

### 7.1 가설
> 만약 wordtest.xlsx 기반의 자동 레벨 테스트를 제공하면, 선생님의 레벨 판정 시간이 80% 이상 줄어들 것이다. 왜냐하면 수동 채점과 기록이 완전히 자동화되기 때문이다.

### 7.2 실험 루프

```
실험 → 관측 → 학습 → 다음 가설
```

| 단계 | 내용 |
|------|------|
| 실험 | 5명의 학생에게 온라인 레벨 테스트 진행 |
| 관측 | 테스트 완료율, 소요 시간, 레벨 판정 정확도 |
| 학습 | 완료율 80% 이상이면 성공, 미만이면 UI/문제 수 조정 |
| 다음 | 학습 모드(FEAT-3) 추가 시 재방문율이 오르는지 검증 |
